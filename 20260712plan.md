# 大宝小宝积分系统 — 亲子积分使用场景评审与优化方案

**评估日期：** 2026-07-12

---

## 一、场景匹配度评审

### 1.1 当前系统定位

项目名"大宝小宝积分系统"明确指向亲子积分管理场景，但当前实现采用**工厂管理模式**（ADMIN = 管理员, WORKER = 工人），与家庭亲子场景存在语义和功能错位。

### 1.2 已覆盖的功能 ✅

| 场景 | 当前支持度 | 说明 |
|------|-----------|------|
| 家长创建任务 | ✅ | 管理员可在管理端创建项目（如"收拾房间"、"完成作业"） |
| 分配任务给孩子 | ✅ | 管理员将项目分配给工人（孩子） |
| 孩子完成任务后提交 | ✅ | 工作端有"完成任务"入口，提交后状态为待审核 |
| 家长审核确认 | ✅ | 管理端审核列表可批准/拒绝，通过后积分入账 |
| 孩子赚取积分 | ✅ | PointsTransaction 记录 EARN 类型 |
| 孩子消耗积分兑换奖励 | ✅ | 消耗类任务存在，提交后自动扣除积分 |
| 积分余额查询 | ✅ | 工作端有积分明细页面 |
| 积分交易记录 | ✅ | 管理端和工作端均可查看过滤后的记录 |
| 管理员手动调整积分 | ✅ | 可正向/负向调整 |

### 1.3 缺失或有问题的功能 ❌

| 场景 | 问题严重度 | 说明 |
|------|-----------|------|
| **角色命名不亲切** | 🟡 中 | "管理员"/"工人"对儿童不友好，应改为"家长"/"宝贝" |
| **无家庭关系模型** | 🔴 严重 | 没有"家庭"概念，一个家长无法管理多个孩子并查看所有孩子的积分 |
| **孩子消耗无需家长审批** | 🔴 严重 | 当前消耗积分直接生效（`submitConsume` 无需审核），孩子可随意花积分 |
| **无奖励清单/兑换商城** | 🟡 中 | 消耗类任务由管理员创建，孩子无法主动浏览可兑换的奖励 |
| **无攒钱目标/愿望清单** | 🟡 中 | 孩子无法设置积分目标（如"攒够200分换玩具"） |
| **无通知提醒** | 🔴 严重 | 任务过期、审核结果、积分变动无任何通知（Web/消息） |
| **无每周/每月报告** | 🟡 中 | 家长无法收到孩子的积分周报/月报 |
| **无打卡/连续签到激励** | 🟢 低 | 无 gamification 元素（连续完成奖励、徽章等） |
| **界面非儿童友好** | 🟡 中 | 界面为通用管理风格，无大按钮、彩色、图标等儿童友好设计 |
| **多个孩子无法并排对比** | 🟡 中 | 家长无法在一个页面看到所有孩子的积分和任务完成情况 |
| **无自定义积分规则** | 🟢 低 | 如额外奖励、惩罚规则等 |

### 1.4 安全问题在亲子场景下的特殊风险

| 问题 | 位置 | 家庭场景影响 |
|------|------|-------------|
| **CSRF 完全禁用** | `SecurityConfig.java:62` | 孩子可能在浏览其他网站时被诱导发起恶意请求 |
| **用户名枚举漏洞** | `UserService.java:234-239` | 暴露家庭用户信息 |
| **密码字段 JSON 泄露** | `User.java` (使用 @Data) | 若孩子使用的接口返回用户信息，密码可能泄露 |
| **余额扣减非原子操作** | `TaskService.java:102-118` | 高并发下超额消耗，但家庭场景影响较小 |
| **无密码强度策略** | `SecurityConfig.java` | 孩子可能设置弱密码 |

## 二、亲子场景优化方案（按优先级排序）

### 第一阶段：角色重塑与关系模型（P0）

#### 2.1 角色模型重构

```
当前: ADMIN = "管理员", WORKER = "工人"
改为: PARENT = "家长", CHILD = "宝贝"
```

在 `Role.java` 中新增 `PARENT` 角色，将原 `ADMIN` 逻辑映射到 `PARENT`，原 `WORKER` 映射到 `CHILD`。同时新增 `family_id` 或 `parent_id` 字段关联亲子关系。

**User 实体新增字段：**

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "parent_id")
private User parent;  // 孩子的家长（仅 CHILD 角色使用）
```

#### 2.2 家长总览仪表盘

新增 `GET /parent/dashboard` 页面，展示：
- 所有孩子的积分余额卡片排布
- 每位孩子的近期活动摘要
- 待审核的任务数量角标

### 第二阶段：核心流程完善（P1）

#### 2.3 消耗积分审批流程

**当前问题：** `submitConsume()` 无需审核直接扣分。

**改造方案：** 消耗积分改为与赚取积分对等的"提交 → 家长审核 → 入账"流程。

```
孩子: 选择奖励 → 提交兑换申请 → 状态 PENDING
家长: 审核 → APPROVED（扣分） / REJECTED（驳回）
```

新增枚举 `ConsumeStatus` 或复用 `TaskExecutionStatus` 的 PENDING/APPROVED/REJECTED。

#### 2.4 奖励清单/兑换商城

新增页面 `GET /parent/rewards` 和 `GET /child/rewards`：

```java
// 新增 Reward 实体
@Entity
@Table(name = "rewards")
public class Reward {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(nullable = false)
    private String name;           // 奖励名称（如"看30分钟动画片"）
    
    @Column(nullable = false)
    private Integer costPoints;    // 所需积分
    
    @Column(length = 500)
    private String description;    // 说明
    
    @Column(name = "image_url")
    private String imageUrl;       // 可选图片
    
    @Enumerated(EnumType.STRING)
    private RewardStatus status;   // ACTIVE / DISABLED
    
    @Column(name = "created_by")
    private Long createdBy;        // 家长创建
    
    @Column(name = "max_per_day")
    private Integer maxPerDay;     // 每日限兑次数（可选）
}
```

孩子端展示所有 ACTIVE 奖励，点击"兑换"发起申请。

#### 2.5 攒钱目标 / 愿望清单

```java
@Entity
@Table(name = "savings_goals")
public class SavingsGoal {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;             // 孩子
    
    @Column(nullable = false)
    private String name;           // 目标名称
    
    @Column(nullable = false)
    private Integer targetPoints;  // 目标积分
    
    private String icon;           // 图标
    
    @Column(name = "is_completed")
    private Boolean completed;
}
```

孩子端展示进度条（当前积分 / 目标积分），达成后家长可手动标记完成。

### 第三阶段：通知与报告（P2）

#### 2.6 通知机制

- 当前无任何通知
- 建议方案（按复杂度递增）：
  1. **页面内通知**：新增 `Notification` 实体，用户登录后在导航栏显示角标
  2. **家长端**：任务待审核、孩子积分变动通知
  3. **孩子端**：审核通过/拒绝、积分到账通知

```java
@Entity
@Table(name = "notifications")
public class Notification {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;             // 接收者
    
    @Column(nullable = false)
    private String title;
    
    private String content;
    
    @Column(name = "is_read")
    private Boolean isRead = false;
    
    @Column(name = "created_at")
    private LocalDateTime createdAt;
}
```

#### 2.7 积分报告

- 家长端每周自动汇总：各孩子赚/扣积分趋势
- 使用现有 `DailyStatsResponse` 数据，新增 `GET /parent/report/weekly` 接口

### 第四阶段：体验优化（P3）

#### 2.8 界面儿童友好化

- 孩子端使用大按钮、明快色彩、Emoji 图标
- 积分余额以可视化方式展示（进度条、卡通数字）
- 任务完成用动画反馈

#### 2.9 打卡与连续激励

- 连续完成任务 X 天获得额外积分奖励
- 在 `RepeatType` 基础上增加 `STREAK_BONUS` 配置

#### 2.10 多孩子对比

- 家长端并列展示所有孩子的积分、完成率
- 排行榜（可选，需注意心理影响）

## 三、基础代码质量修复（亲子场景之外也要修）

此部分已在 `20260712.md` 中详细列出，这里仅汇总亲子场景下**必须优先修复**的部分：

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P0 | 用户名枚举漏洞 | 暴露家庭用户信息 |
| P0 | CSRF 禁用 | 孩子或被诱导执行恶意操作 |
| P0 | 密码字段泄露风险 | 孩子密码可能暴露 |
| P1 | 所有 Request DTO 零校验 | 空值可能引起程序异常 |
| P1 | 精度丢失 (`BigDecimal.intValue()`) | 积分计算可能少算 |
| P2 | `System.out.println` 打印默认密码 | 启动日志泄露家长默认密码 |
| P2 | H2 Console 生产环境可用 | 数据库可被直接访问 |

## 五、总结

**总体评语：** 当前系统具备亲子积分管理的**基础骨架**（任务创建、分配、完成、审核、积分流转），但直接给孩子和家长使用会有**角色感错位、安全风险、功能缺失**三大问题。核心差距在于：角色模型是工厂导向而非家庭导向，消耗流程缺少家长控制，以及缺少家庭场景特有的激励和目标管理功能。

如果仅用于**家长完全管控**的场景（即孩子只完成任务，不自行操作），当前系统勉强可用。但如果要让孩子**自主操作**（浏览任务、查看积分、兑换奖励），则需要按上述方案进行系统改造。

**改造后覆盖度预估：** 完成四个阶段改造后，亲子积分使用场景覆盖度可从当前的 50% 提升至 95% 以上。
